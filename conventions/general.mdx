---
title: "General naming rules for Yarn mappings"
description: "Covers casing, spelling, conciseness, abbreviations, package names, and consistency patterns used across all Yarn name contributions."
---

Yarn follows a consistent set of naming rules to keep mappings readable, memorable, and aligned with the broader Minecraft modding ecosystem. Every name you contribute — whether a class, method, field, or parameter — should satisfy all of the rules on this page before you open a pull request.

## Casing

| Kind | Convention | Example |
|---|---|---|
| Classes and interfaces | `UpperCamelCase` | `ChestBlockEntity` |
| Methods, variables, non-static fields | `lowerCamelCase` | `getCarversForStep` |
| Static final fields (constants) | `UPPER_SNAKE_CASE` | `MAX_BUILD_HEIGHT` |

## Natural language order

Names should read in natural language order. A class that represents a chest block entity is `ChestBlockEntity`, **not** `BlockEntityChest`. Prefix ordering may look tidy in an IDE file tree, but code is read and written far more often than it is browsed.

```java
// Correct — natural language order
class ChestBlockEntity { }
class FurnaceBlockEntity { }

// Incorrect — prefix ordering
class BlockEntityChest { }
class BlockEntityFurnace { }
```

## Spelling

Use American English throughout, consistent with Yarn's existing names and known Mojang names. When two spellings are both acceptable, check whether one is already used in Yarn or by Mojang and prefer that. If neither is established, use the most common spelling.

<Note>
Examples of American vs. British spelling: "color" not "colour", "armor" not "armour", "center" not "centre".
</Note>

## Conciseness

Omit words that are made redundant by the parameter type, owner class name, or call site. Don't avoid overloading methods or shadowing fields.

```java
// Correct — "AtPosition" is redundant given the BlockPos parameter
world.getChunk(BlockPos pos)

// Incorrect — restates information already visible at the call site
world.getChunkAtPosition(BlockPos pos)
```

```java
// Correct — "Box" is redundant on a static method of class Box
Box.create(...)

// Incorrect — redundant class name in method name
Box.createBox(...)
```

Conciseness is a secondary concern to accuracy. A name must first be descriptive — always look at all usages of a member, including overriding methods and inheriting classes, before shortening it. Names used frequently throughout the codebase benefit most from brevity; obscure names can afford to be longer.

## Abbreviations

Avoid abbreviations unless they are well-known to all contributors and are already used consistently in Yarn. Full names are easier to read quickly and do not slow down typing thanks to IDE autocompletion.

The approved abbreviations are:

| Abbreviation | Stands for |
|---|---|
| `id` | identifier |
| `pos` | position |
| `nbt` | named binary tag |
| `init` | initialize |
| `min` / `max` | minimum / maximum |
| `json`, `html`, etc. | abbreviations already used by Java or common libraries |
| `o` | the parameter of `equals(Object)` methods |

### Acronyms as single words

Treat acronyms as single words rather than capitalizing every letter. This improves readability and is consistent with Mojang's own naming style.

```java
// Correct — acronym treated as a single word
JsonObject   // not JSONObject
NbtIo        // not NBTIo (matches the known Mojang name)
```

## Package names

Package names must always be **singular** to follow Java conventions (e.g., `net.minecraft.block`, not `net.minecraft.blocks`). Try to respect Mojang's existing package structure to avoid access-visibility problems when Mojang reorganises code in future versions.

## Consistency

When multiple reasonable names exist, prefer the term already used elsewhere in Yarn, in Minecraft's standard libraries, or in vanilla strings. The sections below list the specific name patterns you must follow.

<AccordionGroup>

<Accordion title="Ticks and updates">

Use **"tick"** for operations that run exactly once per game tick. Use **"update"** for other kinds of state changes.

```java
void tick() { }          // runs once per game tick
void updateNeighbors() { } // triggered by state changes, not the tick loop
```

</Accordion>

<Accordion title="Value from the previous tick">

Use the prefix **"last"** for fields that store the value a property held during the previous tick.

```java
double lastX;
double lastWidth;
double lastHealth;
```

</Accordion>

<Accordion title="Getters, setters, withers, and creators">

| Prefix | Usage |
|---|---|
| `get` | Non-boolean getter; methods that compute a property with no side effects beyond caching in a private field. |
| `is` | Boolean getter. |
| `set` | Mutating setter. Name the parameter after the property itself — `setColor(color)`, not `setColor(newColor)`. |
| `with` | Returns a copy of the object with one property changed. Name the parameter after the property. |
| `create` | Creates a new instance of an object. |
| `getOrCreate` | Creates a new instance only if one does not already exist. Do not use for lazy initialisation. |

</Accordion>

<Accordion title="Serialization">

| Term | Usage |
|---|---|
| `serializer` | An object whose purpose is serialising or deserialising another type (`RecipeSerializer`). |
| `serialize` / `deserialize` | Methods that serialise or deserialise an object *other than* the one the method belongs to. |
| `from` | Static factory that creates an instance of the owner type from external data — `fromJson`, `fromNbt`, `fromString`. |
| `to` | Instance method that converts the object to another type — `toString`, `toLong`, `toNbt`. |
| `read` | Non-static method that loads data into the current object. |
| `write` | Method that saves data to an *existing* object passed as a parameter. |

</Accordion>

<Accordion title="Factories and builders">

Use **"factory"** for objects whose purpose is creating other objects.

Use **"builder"** for objects that assist with constructing immutable objects. Name builder setter methods the same as the field they set, with no prefix:

```java
// Builder method — no "set" prefix, matches the field name
builder.color(color)      // not builder.setColor(color)
builder.hardness(2.0f)    // not builder.setHardness(2.0f)
```

</Accordion>

<Accordion title="Collections">

Use a **plural noun** for collections and maps rather than including the words "list", "set", or "array" in the name, unless you need to distinguish between multiple collections of different types holding the same objects.

```java
List<Entity> entities;      // correct — plural noun
List<Entity> entityList;    // only correct when there is also, e.g., a Set<Entity>
```

Name maps by their value type when that is sufficient. When the key type also needs to be explicit, use the **`valuesByKeys`** format:

```java
Map<String, Block> blocks;             // named by value type
Map<Block, String> translationsByBlock; // named using valuesByKeys format
```

</Accordion>

<Accordion title="Coordinates">

Use `x`, `y`, and `z` directly when the context makes their meaning clear. When clarification is needed, prepend a descriptive word **before** the coordinate letter:

```java
double velocityX;   // correct — word before coordinate
double xVelocity;   // incorrect — coordinate before word
```

For screen coordinates, use `x` and `y` rather than `left` and `top`.

</Accordion>

</AccordionGroup>

## Special classes

Certain Minecraft classes appear so frequently as parameters and fields that Yarn has standardised names for them. Use these names unless a more descriptive name better communicates intent in context.

| Class | Standard name |
|---|---|
| `net.minecraft.client.render.VertexConsumerProvider` | `vertexConsumers` |
| `net.minecraft.client.util.math.MatrixStack` | `matrices` |
| `net.minecraft.registry.RegistryWrapper.WrapperLookup` | `registries` |

## Things to avoid

**Do not name methods based on implementation details.** Names should describe what a method does, not how it is implemented. A name that exposes implementation details becomes misleading the moment the implementation changes.

**Do not include Java-specific information in names.** Avoid:

- Prefixing interfaces with `I` (e.g., `IBlock`)
- Prefixing enums with `Enum` (e.g., `EnumDirection`)
- Prefixing methods to indicate visibility (e.g., `privateUpdate`)

Instead, find a meaningful name that describes the difference between classes. For abstract classes, this may mean renaming subclasses to more specific names rather than marking the abstract class with a prefix.
